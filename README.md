# Title: ICMP-Based Network Diagnostic Suite with Ping and Traceroute

## Abstract

This mini-project is about creating a simple network diagnostic tool using Internet Control Message Protocol (ICMP). 

The main idea is to create a tool that combines Ping and Traceroute to test the performance of a network by sending ICMP packets to a
destination. It uses raw ICMP sockets to send Echo Request and Echo Reply packets, which allow users to measure the Round Trip Time (RTT) and trace the route that packets take to reach a destination.

---

## Architecture

C ICMP Program → Output Files → Dashboard (Parser + UI)

The C program generates ICMP diagnostic data, which is stored in text files.
The dashboard reads these files, extracts relevant information, and presents it visually using graphs and structured components.

---

## Implementation Details

The program is implemented in C using raw sockets to create and send ICMP packets.

* The **ping operation** sends ICMP echo requests and receives replies to measure round-trip time (RTT) and packet loss.
* The **traceroute operation** increases the Time-To-Live (TTL) value step by step to identify each intermediate hop between the source and destination.

---

## Features

* ICMP Ping
* ICMP Traceroute
* RTT Measurement
* Packet Loss Calculation
* Hop-by-hop Route Discovery
* Dashboard Visualization
* Raw Output Display

## How to Run

### 1. Compile the program

```bash
gcc icmp_diag.c -o icmp_diag
```


### 2. Generate output files

 ```bash
    sudo ./icmp_diag ping google.com > ping_output.txt
 ```

  ```bash
     sudo ./icmp_diag traceroute google.com > traceroute_output.txt
```

### 3. Make sure  the following files are in the same folder

* icmp_diag.c
* icmp_diag
* ping_output.txt
* traceroute_output.txt
* dashboard.html

### 4. Start the local server

```bash
   python3 -m http.server 8000
```

### 5. Open the dashboard

```bash
   http://localhost:8000/dashboard.html 
```

## Performance Evaluation

From the executed results:

* Average RTT for google.com is approximately **30 ms**
* Packet loss is **0%**, indicating a stable connection
* Traceroute reaches the destination in approximately **11 hops**
* Some intermediate hops may show higher latency due to routing or temporary network conditions

### Observations

* Lower RTT indicates faster response time
* Zero packet loss indicates stable connectivity
* Higher number of hops indicates a longer routing path

Overall, the network performance observed was stable with low latency and no packet loss.
